# 実装ロードマップ

## 前提・注意事項

- `main` ブランチへのマージ = **Vercel本番に即時反映**
- 各フェーズは独立してマージ可能な単位で設計
- 認証・RLSが整うまでは、本番に個人データが漏れる状態を作らない
- フェーズ内のタスクは上から順に実施する

## Phase 12.9: 小テスト JSON インポート機能（割り込み）

> AI で生成した JSON を貼り付けるだけで小テストを一括作成できる機能を追加する。
> 現状は1問ずつ手入力する必要があるため、AI 生成との連携ワークフローを大幅に改善する。

### 背景

教師が Claude などの AI に問題を生成させ、その出力 JSON を直接小テスト作成フォームに流し込める仕組みを求めている。

### JSON フォーマット仕様

AI に渡すプロンプト・インポート両方でこの形式を使う：

```json
{
  "title": "クイズタイトル",
  "questions": [
    {
      "type": "multiple_choice",
      "content": "問題文（プレーンテキスト）",
      "explanation": "解説（省略可、空文字でも可）",
      "options": ["選択肢A", "選択肢B", "選択肢C", "選択肢D"],
      "correctAnswerIndex": 0
    },
    {
      "type": "short_answer",
      "content": "問題文",
      "explanation": "解説",
      "correctAnswerText": "模範解答"
    },
    {
      "type": "ordering",
      "content": "問題文",
      "explanation": "解説",
      "options": ["正しい順の要素1", "要素2", "要素3"]
    }
  ]
}
```

### 実装方針

- `QuizForm` に「JSON で一括インポート」ボタンとテキストエリアを追加
- JSON を貼り付けて「インポート」を押すと、既存フォームの `questions` ステートに変換してセットする
- `content` / `explanation` は プレーンテキスト → tiptap JSON（`paragraph` ノード）に変換
- インポート後は通常の手入力フォームと同じ状態になるため、内容を確認・編集してからそのまま保存できる
- パースエラー時は toast でエラー表示（フォームの状態は変更しない）

### タスク

- [ ] `QuizForm` に JSON インポートパネルを追加（折りたたみ式 UI）
  - テキストエリアに JSON を貼り付けて「インポート」ボタンを押すとフォームに反映
  - タイトルも JSON から自動セット
- [ ] プレーンテキスト → tiptap JSON 変換ユーティリティを実装

### マージ判断

インポート後に内容を確認・編集して保存できること、パースエラーが適切に処理されることを確認してからマージ

- いいね数が可視化されると「受けを狙った投稿」が増え、正直な気づきが書きにくくなる
- 過激な内容ほど注目を集めるというSNS的な歪みが生まれるリスクがある
- 代替として Phase 17 の匿名プライベートコメント機能で「共有したことが意味を持った感覚」を実現する

## Phase 15: 視聴完了ゲート

> 動画を見終わるまで他の生徒の投稿を非表示にし、動画への集中と自分なりの思考形成を保護する。

### 設計方針

- 動画を最後まで再生するまで、投稿一覧はロック状態で表示しない
- 視聴完了後にアンロックされ「他の人の視点を見てみよう」という自然な流れを作る
- 飛ばし見は許容する（生徒を信頼する）。完了判定は YouTube Player の `onEnd` イベントを使用
- 視聴完了状態はセッション内で保持（ページを一度離れても再視聴不要にするか要検討）

### タスク

- [ ] `VideoPlayer` で `onEnd` イベントを検知し、親コンポーネントに完了状態を通知する
- [ ] 視聴完了前は `PostList` をロック表示（「動画を最後まで見ると、みんなの投稿が見られます」などのメッセージ）
- [ ] 視聴完了時に投稿一覧がアンロックされるUIを実装
- [ ] 視聴完了フラグをコンポーネントのstateで管理（セッション内保持）

### マージ判断

動画を最後まで再生しないと投稿一覧が見えないことを確認してからマージ

## Phase 16: トロフィー・実績機能

> 外発的動機付けを足場として、内発的動機付けへの移行を促す。自分の成長を可視化し、自己肯定感を育てる。

### 設計方針

- トロフィーは**本人にのみ表示**（他の生徒には非公開・競争させない）
- 「量」ではなく「最初の一歩」「習慣の形成」を祝う設計
- 自分の過去と比較する形式（他者比較ではなく自己比較）
- `/profile` ページに「学習の足跡」セクションとして表示
- 最初のトロフィーはできるだけ早く・簡単に取れるように設計する

### トロフィー種別（案）

| トロフィー名 | 付与条件 |
|---|---|
| 初メモ | はじめてメモを保存した |
| 初シェア | はじめて投稿をクラスに共有した |
| 探究者 | 3つ以上の異なるレッスンでメモを書いた |
| 継続学習者 | 連続するレッスンでメモを書いた |
| 外部リンク貢献 | メモに外部URLを含めて投稿した |
| 再挑戦 | 同じ小テストに2回以上チャレンジした |

### タスク

- [ ] `achievements` テーブルを追加（`user_id`, `type`, `earned_at`）
- [ ] マイグレーションファイル作成
- [ ] トロフィー付与ロジックを各アクション時に組み込む（メモ保存・投稿・小テスト回答時）
- [ ] `GET /api/achievements` — 自分のトロフィー一覧を取得するAPI
- [ ] `/profile` ページにトロフィー表示セクションを追加
- [ ] RLS設定（本人のみ閲覧可）

### マージ判断

本人以外にトロフィーが見えないこと、各トロフィーが正しい条件で付与されることを確認してからマージ

## Phase 17: 匿名プライベートコメント機能

> 投稿した生徒に「共有したことが意味を持った」という感覚を届ける。SNS的な公開スレッドではなく、匿名の私的なメッセージとして設計する。

### 設計方針

- コメントは**投稿した本人にのみ届く**（公開コメントスレッドではない）
- コメントした側は**匿名**（投稿者にはコメントした人が誰かわからない）
- コメントを書いた本人にも**自分のコメントはUIに残らない**（社会的アピールに使えない構造）
- ただし**DBには送信者IDを記録**（モデレーション・不正調査のため）
- AIによる**事前モデレーション**（届ける前に判定し、問題あればブロック）
- 受け取った生徒が**フラグを立てる**ことで教師が確認・送信者を特定できる

### モデレーション設計

```
コメント送信
  ↓
一次フィルタ（明らかに問題ない → 通過）
  ↓
グレーゾーン → AI判定（LLM API呼び出し）
  ↓
問題あり → ブロック・DB記録・教師に通知
問題なし → 投稿者へ届ける
```

- 隠語・スラング・新しい表現への対応のためLLMを使用
- 全コメントにAI呼び出しをするのではなく段階的フィルタでコストを抑える

### タスク

- [ ] `post_comments` テーブルを追加
  - `id`, `post_id`（FK）, `sender_id`（FK・UI非表示）, `content`（text）, `is_blocked`, `is_flagged`, `created_at`
- [ ] マイグレーションファイル作成
- [ ] RLS設定（投稿者のみ自分の投稿へのコメントを閲覧可・送信者はUIから特定不可）
- [ ] `POST /api/posts/[postId]/comments` — コメント投稿API（AIモデレーション込み）
- [ ] `GET /api/posts/[postId]/comments` — コメント一覧取得API（投稿者のみ）
- [ ] `POST /api/posts/[postId]/comments/[commentId]/flag` — フラグ報告API
- [ ] 投稿カードに「コメントが届きました」通知UIを追加（送信者情報なし）
- [ ] コメント閲覧UI（投稿者のみ表示・誰からかは非表示）
- [ ] コメント入力UI（送信後は入力欄のみリセット、送信済みコメントは表示しない）
- [ ] 教師用：フラグされたコメントを確認・送信者を特定できる管理画面

### マージ判断

- コメントが投稿者以外に見えないこと
- 送信者が特定できないこと（UIレベル）
- 問題のあるコメントがブロックされること
- フラグ報告が教師に届くこと

を確認してからマージ
