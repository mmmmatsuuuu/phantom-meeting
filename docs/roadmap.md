## 前提・注意事項

- `main` ブランチへのマージ = **Vercel本番に即時反映**
- 各フェーズは独立してマージ可能な単位で設計
- 認証・RLSが整うまでは、本番に個人データが漏れる状態を作らない
- フェーズ内のタスクは上から順に実施する

## Phase 12.9: 小テスト JSON インポート機能（割り込み）

> AI で生成した JSON を貼り付けるだけで小テストを一括作成できる機能を追加する。
> 現状は1問ずつ手入力する必要があるため、AI 生成との連携ワークフローを大幅に改善する。

### 背景

教師が Claude などの AI に問題を生成させ、その出力 JSON を直接小テスト作成フォームに流し込める仕組みを求めている。

### JSON フォーマット仕様

AI に渡すプロンプト・インポート両方でこの形式を使う：

```json
{
  "title": "クイズタイトル",
  "questions": [
    {
      "type": "multiple_choice",
      "content": "問題文（プレーンテキスト）",
      "explanation": "解説（省略可、空文字でも可）",
      "options": ["選択肢A", "選択肢B", "選択肢C", "選択肢D"],
      "correctAnswerIndex": 0
    },
    {
      "type": "short_answer",
      "content": "問題文",
      "explanation": "解説",
      "correctAnswerText": "模範解答"
    },
    {
      "type": "ordering",
      "content": "問題文",
      "explanation": "解説",
      "options": ["正しい順の要素1", "要素2", "要素3"]
    }
  ]
}
```

### 実装方針

- `QuizForm` に「JSON で一括インポート」ボタンとテキストエリアを追加
- JSON を貼り付けて「インポート」を押すと、既存フォームの `questions` ステートに変換してセットする
- `content` / `explanation` は プレーンテキスト → tiptap JSON（`paragraph` ノード）に変換
- インポート後は通常の手入力フォームと同じ状態になるため、内容を確認・編集してからそのまま保存できる
- パースエラー時は toast でエラー表示（フォームの状態は変更しない）

### タスク

- [x] `QuizForm` に JSON インポートパネルを追加（折りたたみ式 UI）
  - テキストエリアに JSON を貼り付けて「インポート」ボタンを押すとフォームに反映
  - タイトルも JSON から自動セット
- [x] プレーンテキスト → tiptap JSON 変換ユーティリティを実装

### マージ判断

インポート後に内容を確認・編集して保存できること、パースエラーが適切に処理されることを確認してからマージ

  - 第2段階：生徒行をクリック → その生徒のメモのみをオンデマンドで取得・展開（アコーディオン）
- 初回ロードは約40件のメタデータのみ。全件一括取得は行わない

## Phase 17: 匿名プライベートコメント機能

> 投稿した生徒に「共有したことが意味を持った」という感覚を届ける。SNS的な公開スレッドではなく、匿名の私的なメッセージとして設計する。

### 設計方針

- コメントは**投稿した本人にのみ届く**（公開コメントスレッドではない）
- コメントした側は**匿名**（投稿者にはコメントした人が誰かわからない）
- コメントを書いた本人にも**自分のコメントはUIに残らない**（社会的アピールに使えない構造）
- ただし**DBには送信者IDを記録**（モデレーション・不正調査のため）
- AIによる**事前モデレーション**（届ける前に判定し、問題あればブロック）
- 受け取った生徒が**フラグを立てる**ことで教師が確認・送信者を特定できる

### モデレーション設計

```
コメント送信
  ↓
一次フィルタ（明らかに問題ない → 通過）
  ↓
グレーゾーン → AI判定（LLM API呼び出し）
  ↓
問題あり → ブロック・DB記録・教師に通知
問題なし → 投稿者へ届ける
```

- 隠語・スラング・新しい表現への対応のためLLMを使用
- 全コメントにAI呼び出しをするのではなく段階的フィルタでコストを抑える

- **集計粒度**: **学年全体の集計 ＋ クラス別内訳を横並び**（クラス絞り込みではなく全体俯瞰）
- **誤答内訳の表示条件**: 正答率 **60% 以下**の設問のみ（それ以上は理解が定着していると見なす）
- **記述式サンプル**: 最新回答からランダム3件を抽出（正誤判定なし・学籍番号なし）

#### エクスポートフォーマット（CSV）

プロンプト（文脈・依頼文）とCSVファイル（データ）を分離し、AIへの役割分担を明確にする。
セクション見出し（`##` 行）を使い、1ファイルに3種のデータを格納する。

### タスク (c): メモエクスポート

> 単元内の各レッスンで生徒が残したメモをサンプリングして CSV でエクスポートし、生徒の理解状況・興味関心を俯瞰する材料を提供する。
> 小テスト結果と合わせて AI に渡すことで、より多角的な授業改善が可能になる。

#### 集計ルール

- **集計対象**: 単元内の全レッスン × 学年全体の生徒が書いたプライベートメモ（`memos` テーブル）

### タスク (b): 生徒へのフィードバック

> 小テスト結果をもとに生徒一人ひとりへの個別フィードバックを届ける手段を提供する。

#### 運用形式の選択肢

**案1：CSV出力による手動連携**（タスク(a)と同じアプローチ）
- 生徒別の小テスト結果をCSVで出力し、教師が外部AIで分析してフィードバックを作成
- 実装コストが低く、追加のAPI連携が不要
- フィードバックの配布手段は別途検討（既存の備考欄活用など）

**案2：システム内での自動完結**（外部AI APIとの連携が必要）
- 生徒主導型：生徒がボタンを押すとその場でAIアドバイスが返ってくる
- 教師主導型：教師が任意のタイミングで一括集計を実行し、AIフィードバックを各生徒の画面に表示

#### 案2の実装前に確認すべき課題

- **課金体系の確認**：現在の個人契約ライセンス（Claude等）で対応可能か、別途API利用料が発生するかを検討する
- **利用制限の設計**：コスト管理のため、単元ごとに1回まで等の回数制限を設ける仕組みが必要

#### 方針

案1を先行実装し、運用しながら案2の費用対効果を見極める。

### マージ判断

teacher ロール以外がエクスポートできないこと、出力データが学籍番号での識別にとどまり氏名・メールアドレス等の個人情報を含まないことを確認してからマージ
